## 2. 关键设计原则

### 2.1 插件独立性

**核心原则**：插件之间不允许有依赖关系，每个插件必须是独立完整的数据采集工具。

**治理要求**：
- 依赖声明：必须为 empty
- 故障隔离：单个插件失效不影响其他
- 部署独立：单个文件即完整插件
- 版本独立：每个插件独立版本迭代

### 2.2 上游开放（SourceEvent 接入）

**核心原则**：DCP 只是其中一个上游采集平台，系统通过标准化的 SourceEvent 协议接入任意上游系统。

**设计要点**：
- 统一接入协议：`/ingestion/v1/events` 接收标准化 SourceEvent
- 幂等性保障：通过 `idempotency_key` 防止重复接入
- 数据溯源：保留 `source_system`、`source_record_key` 等溯源字段
- 灵活扩展：支持任意上游系统，不限于 DCP

### 2.3 下游开放（Sandbox API）

**核心原则**：数字沙盘只是其中一个下游应用，系统通过 Sandbox API 为任意下游应用提供专用数据接口。

**设计要点**：
- 专用接口：为特定下游应用提供定制化数据查询
- 数据隔离：Sandbox API 返回标准化 DTO，不暴露原始 DCP 字段
- 灵活扩展：可新增其他下游应用专用 API

### 2.4 逻辑隔离（非进程级）

**当前实现**：协程级隔离

| 隔离类型 | 实现方式 | 说明 |
|----------|----------|------|
| 配置隔离 | 每个插件独立配置 | 不共享配置对象 |
| 实例隔离 | 每次create_adapter()新实例 | 无单例缓存 |
| 错误边界隔离 | try-except捕获异常 | 单个插件异常不扩散 |

**注意**：协程级隔离 ≠ 进程级强隔离。插件卡死或崩溃仍可能影响主服务。

**后续考虑**：子进程执行方案

### 2.5 延迟加载

**问题**：`discover_plugins() → import所有模块`，插件多时会爆炸

**解决方案**：

discover阶段:
  - 扫描文件
  - AST解析提取元数据
  - 不执行import

create_adapter阶段:
  - 真正import模块
  - 创建实例

**收益**：
- 启动快：只解析不执行
- 内存省：不用的插件不加载

### 2.6 受控执行模型

**设计目标**：防止单个插件卡死或崩溃影响整个系统。

**控制机制**：

| 控制维度 | 机制 | 默认值 | 说明 |
|----------|------|--------|------|
| 并发控制 | 信号量（Semaphore） | 2 | 限制同时执行的任务数 |
| 超时控制 | 任务超时限制 | 30秒 | 单个任务最大执行时间 |
| 异常隔离 | 错误边界捕获 | - | 单个任务异常不扩散 |

**执行流程**：
1. 调度器触发任务
2. 获取信号量（控制并发）
3. 执行任务（带超时限制）
4. 释放信号量
5. 异常处理（记录日志，不影响其他任务）

### 2.7 单轮询广播模式

> 📄 **详细设计**：[ADR-003 WebSocket单轮询广播模式](./decisions/ADR-003-single-poll-broadcast-websocket.md)

**问题**：100个WebSocket客户端 → 100个轮询 → SQLite直接炸

**核心机制**：
- 单轮询：一个后台任务定期查询数据库
- 广播：将新数据推送给所有连接的客户端
- 过滤：支持客户端自定义过滤条件

**收益**：无论多少客户端，只有一个数据库查询，支持100+并发连接

## 3. 核心组件设计

### 3.1 PluginManager - 延迟加载

**问题**：插件多时，一次性import所有模块导致启动慢、内存占用高。

**解决方案**：

| 阶段 | 行为 | 资源占用 |
|------|------|----------|
| **发现阶段** | AST解析提取元数据，不执行import | 低 |
| **使用阶段** | 真正import模块并创建实例 | 按需 |

**关键设计**：
- 元数据缓存：插件ID、名称、版本、标签等
- 类定义懒加载：首次使用时才import
- 实例隔离：每次create_adapter创建新实例，无单例缓存

### 3.2 TaskScheduler - 受控执行

**核心职责**：
1. 任务调度：基于APScheduler实现定时触发
2. 执行控制：限制并发、超时保护
3. 状态管理：记录执行统计、触发告警

**调度流程**：
1. 接收调度请求（cron表达式）
2. 注册到APScheduler
3. 定时触发执行任务
4. 执行前检查插件启用状态
5. 获取信号量（并发控制）
6. 执行采集（带超时保护）
7. 保存数据、更新统计
8. 释放信号量

**异常处理**：
- 超时：记录超时日志，标记任务失败
- 异常：捕获异常，更新失败统计，检查告警条件

### 3.3 WebSocketManager - 单轮询广播

> 📄 **详细设计**：[ADR-003 WebSocket单轮询广播模式](./decisions/ADR-003-single-poll-broadcast-websocket.md)

**问题**：传统多轮询模式下，N个客户端产生N个数据库查询，导致数据库压力随客户端数量线性增长。

**核心设计**：单轮询 + 广播
- 一个后台任务轮询数据库
- 将新数据广播给所有连接的客户端
- 客户端数量不影响数据库查询频率

**架构**：
```
┌─────────────┐
│  单轮询任务  │◄── 每5秒查询一次数据库
└──────┬──────┘
       │ 广播新数据
       ▼
┌───────────────────────────────────┐
│  所有连接的WebSocket客户端          │
│  [Client1, Client2, ..., ClientN]  │
└───────────────────────────────────┘
```

### 3.4 MCPServer - LLM工具封装层

> 📄 **详细设计**：[ADR-006 MCP Server支持](./decisions/ADR-006-mcp-server-support.md)

**定位**：MCP Server 不是新的数据平面，而是 REST/调度能力的 **LLM工具封装层**。

**核心设计原则**：

| 原则 | 说明 |
|------|------|
| **能力复用** | 复用现有 REST API 和调度能力，不新建业务逻辑 |
| **语义封装** | 将 REST 资源操作转换为 LLM 友好的工具语义 |
| **零侵入** | 插件无需修改，MCP 是系统层封装 |

**复用的系统能力**：

| MCP工具 | 复用的系统能力 | 对应REST端点 |
|---------|---------------|--------------|
| `list_plugins` | PluginManager.get_all_plugins() | `GET /api/plugins` |
| `query_data` | SQLiteStore.query_normalized_data() | `GET /api/data/normalized` |
| `trigger_plugin` | TaskScheduler.trigger_task() | `POST /api/plugins/{id}/trigger` |

**不做的事情**（明确边界）：

| 不做 | 原因 |
|------|------|
| 不直接访问数据库 | 必须通过存储层，保持数据一致性 |
| 不绕开治理逻辑 | 仍需检查插件启用状态、健康状态 |
| 不实现新采集逻辑 | 只触发已有调度能力，不替代 Scheduler |
| 不维护独立状态 | 无 MCP 专用状态，依赖现有系统状态 |

**与 REST API 的关系**：

```
┌─────────────────────────────────────────────┐
│         Data Collector Hub                 │
│                                            │
│   REST API          MCP Server            │
│   (开发者接口)       (LLM工具接口)          │
│        │                │                 │
│        └────────┬───────┘                 │
│                 │                         │
│         ┌───────▼───────┐                 │
│         │  核心引擎层   │                 │
│         │  (复用)       │                 │
│         └───────────────┘                 │
└─────────────────────────────────────────────┘
```

**关键区别**：

| 维度 | REST API | MCP Server |
|------|----------|------------|
| **接口风格** | 资源操作（CRUD） | 工具调用（Tools） |
| **发现机制** | OpenAPI/Swagger | MCP服务发现（`GET /mcp`） |
| **目标用户** | 开发者、系统集成 | LLM/AI Agent |
| **语义** | `/api/data?plugin_id=xxx` | `query_data(plugin_id="xxx")` |

**收益**：
- LLM 可直接理解并调用数据采集能力
- 无需为每个 LLM 应用单独开发适配层
- 标准化工具语义，跨 LLM 平台兼容

### 3.5 NormalizerRunner - 归一化处理

**定位**：将上游系统推送的 `raw_events` 转换为下游应用可消费的 `canonical_entities`。

**核心职责**：
1. 读取 `raw_events` 中未处理的记录
2. 根据 `dataset_key` 选择对应的 normalizer
3. 将原始数据转换为标准化的实体数据
4. 写入 `canonical_entities`（UPSERT 语义）

**处理流程**：
```
1. 读取 normalizer_state 获取上次处理位置
   │
   ▼
2. 查询 raw_events 中 id > last_raw_event_id 的记录
   │
   ▼
3. 按 dataset_key 分组，调用对应 normalizer
   │
   ▼
4. 生成 canonical_entities（UPSERT）
   │
   ▼
5. 更新 normalizer_state
```

**两种执行模式**：

| 模式 | 用途 | 调用方式 |
|------|------|----------|
| **前台运行** | 调试、小规模数据 | `POST /processing/v1/run` |
| **后台作业** | 生产环境大规模数据 | `POST /processing/v1/jobs` |

### 3.6 Ingestion API - 上游数据接入

**定位**：标准化上游数据接入接口，支持任意上游系统推送数据。

**核心设计**：
- 协议：HTTP POST `/ingestion/v1/events`
- 数据格式：SourceEvent 标准格式
- 幂等性：通过 `idempotency_key` 去重
- 批量：支持单次推送多条事件

**SourceEvent 核心字段**：

| 字段 | 说明 |
|------|------|
| `event_id` | 事件唯一标识 |
| `idempotency_key` | 幂等键，防止重复接入 |
| `source_system` | 上游系统名称（如 "dcp"） |
| `source_event_type` | 上游事件类型 |
| `dataset_key` | 数据集标识，用于路由 normalizer |
| `payload` | 原始数据载荷 |
| `occurred_at` | 事件发生时间 |

### 3.7 Sandbox API - 下游应用接口

**定位**：为特定下游应用（如数字沙盘）提供专用数据查询接口。

**核心设计**：
- 数据隔离：返回标准化 DTO，不暴露原始 DCP 字段
- 专用语义：接口命名和数据结构贴合下游应用需求
- 独立演进：Sandbox API 可独立于通用 REST API 演进

**当前接口**：

| 端点 | 说明 |
|------|------|
| `GET /api/v1/sandbox/dates` | 获取可用日期列表 |
| `GET /api/v1/sandbox/map/skeleton` | 获取地图骨架数据 |
| `GET /api/v1/sandbox/map/summary` | 获取地图汇总数据 |

## 4. 数据层架构

> 📄 **详细定义**：[04-data-model.md §1. 数据层架构](./04-data-model.md#1-数据层架构)

数据层采用**多层架构**：
- **raw_data**：插件采集的原始数据
- **raw_events**：上游系统推送的标准化 SourceEvent
- **normalized_data**：插件规范化数据（轻量级提取）
- **canonical_entities**：下游应用消费的实体数据（归一化处理后）

**核心设计原则**：
- `raw_data` 和 `raw_events` 分别服务插件采集和上游接入两条独立数据流
- `normalized_data` 是"半结构化数据层"，在"完全结构化"和"完全灵活"之间取得平衡
- `canonical_entities` 是面向下游应用的标准化实体数据
- 支持渐进式schema演进

## 5. 关键非功能性设计

### 5.1 性能设计

| 设计点 | 实现 | 目标 |
|--------|------|------|
| 延迟加载 | AST解析元数据 | 启动时间 < 3s（100个插件） |
| 并发控制 | Semaphore(2) | 防止资源耗尽 |
| 超时控制 | wait_for(30s) | 防止单任务卡死 |
| 单轮询广播 | 1个查询广播N客户端 | 支持100+并发连接 |
| 数据库索引 | 关键字段索引 | 查询 < 200ms |

### 5.2 可靠性设计

| 设计点 | 实现 | 目标 |
|--------|------|------|
| 错误边界 | try-except隔离 | 单插件失败不影响其他 |
| 重试机制 | APScheduler misfire | 错过执行后补执行 |
| 告警机制 | 连续失败/失败率阈值 | 及时发现异常 |
| 状态持久化 | SQLite + WAL模式 | 数据不丢失 |
| 幂等性 | idempotency_key | 防止重复数据 |

### 5.3 可扩展性设计

| 设计点 | 实现 | 说明 |
|--------|------|------|
| 扁平化插件 | 一个文件一个插件 | 新增插件无代码修改 |
| 标签分类 | tags字段 | 动态组织，无目录限制 |
| 多层数据 | raw/raw_events/normalized/canonical | 渐进式数据处理 |
| 多协议接口 | REST/RSS/WebSocket/MCP/Sandbox | 适配不同下游（含LLM工具） |
| 上游开放 | SourceEvent 标准化协议 | 支持任意上游系统 |
| 下游开放 | Sandbox API | 支持任意下游应用 |
